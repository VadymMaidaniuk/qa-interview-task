# Тестова стратегія

## Що тестувати в першу чергу

На новому проєкті без автоматизованого тестування перший пріоритет — швидко покрити контракти користувача та системи з найвищим ризиком. Я б почав з API-тестів для authentication, CRUD основного ресурсу, validation rules і response schemas, оскільки вони працюють швидше за E2E-тести й падають ближче до джерела проблем бекенду або контракту.

Наступний пріоритет — невеликий набір E2E smoke і critical-path тестів. Вони мають підтвердити, що користувач може увійти в систему, перейти між основними сторінками, відправити важливі форми та побачити в UI дані, що надходять через API. Це дає впевненість, що головні інтегровані частини продукту працюють, не перетворюючи browser suite на повільний дубль API-покриття.

Для цього репозиторію перший автоматизований зріз містить:

- успішну та неуспішну API-авторизацію
- отримання contact list через API із schema validation
- створення контакту через API і перевірку його наявності через list endpoint
- API-перевірки validation failures для обов’язкових полів і некоректного email
- успішний і неуспішний UI login
- UI-навігацію від contact list до add contact і назад
- створення контакту через UI з валідними даними
- валідацію форми в UI для невалідних даних
- відображення в UI контакту, створеного через API

## Розподіл відповідальності між API та E2E

API-тести покривають більшість перевірок контрактів і правил даних. Саме там доречно перевіряти status codes, response schemas, validation messages, response time, authorization rules і CRUD chains. Вони працюють швидше, менш крихкі, ніж browser-тести, і дозволяють покрити більше негативних сценаріїв із меншими витратами на підтримку.

E2E-тести відповідають за інтегровану впевненість у користувацькому сценарії. Вони мають перевіряти, що браузер, routing, forms, authentication cookie, API calls і відмальований UI працюють разом. Вони не повинні вичерпно перевіряти кожну гілку backend validation, якщо API-рівень може зробити це точніше й напряму.

У цьому suite API-тести перевіряють деталі contact contract і негативні сценарії з payload. E2E-тести перевіряють, що характерні позитивні й негативні сценарії роботи форми коректно з’єднані в браузері.

## Стратегія тестових даних

Кожен тест створює власний user account з унікальною email-адресою. Token цього користувача використовується для API setup і cleanup. Контакти створюються в межах цього користувача, тому видалення тестового користувача видаляє й пов’язані з ним дані та спрощує очищення.

Підготовка даних відбувається найдешевшим стабільним шляхом:

- API-тести готують дані й роблять assertions напряму через API calls.
- E2E login-тести створюють користувача через API, а далі проходять login через UI.
- E2E-тести на відображення контактів створюють контактні дані через API, а потім перевіряють, що UI їх відображає.
- E2E-тести форм створюють користувача через API, а UI використовують лише для поведінки, яка тестується.

Це дозволяє уникнути крихкого UI-only сетапу. Наприклад, тест, якому потрібен лише контакт у списку, не повинен витрачати час на проходження add-contact форми через UI, якщо сама форма не є об’єктом перевірки.

Cleanup реалізований у спільній fixture `registeredUser`. Після кожного тесту fixture видаляє поточного користувача. Cleanup толерує вже видалених користувачів, тому навіть якщо акаунт був навмисно видалений у межах тесту, teardown не впаде.

## Як тримати E2E lean

E2E-тести дорогі, тому що одночасно залежать від браузера, мережі, рендерингу UI та стану бекенду. Вони цінні, але лише тоді, коли відповідають на питання, на які нижчі рівні тестів відповісти не можуть.

Стратегія полягає в тому, щоб тримати E2E-тести сфокусованими на кількох користувацьких workflow:

- Чи може валідний користувач увійти в систему?
- Чи відхиляється невалідний login у UI?
- Чи може користувач переходити між важливими сторінками?
- Чи відправляє contact form валідні дані та чи показує validation errors?
- Чи відображаються в contact list дані, що приходять через API?

Перевірки з великою кількістю логіки та даних мають бути на API-рівні. Саме тому schema validation, CRUD read-back, status codes, response time checks і кілька негативних сценаріїв payload покриті в API-тестах. У результаті виходить suite, який простіше дебажити: API-failures зазвичай означають проблеми контракту або поведінки бекенду, тоді як E2E-failures зазвичай вказують на інтеграційні проблеми або помилки в UI-flow.
